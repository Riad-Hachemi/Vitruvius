Built different

🔒

End-to-end encrypted

ChaCha20-Poly1305 authenticated encryption. Keys are derived locally via X25519 Diffie-Hellman — no server ever sees your key material.

🌐

Zero infrastructure

Peers discover each other via mDNS on your LAN. No relay, no STUN, no signup. If they're on the same subnet, they connect.

🤝

TOFU trust model

SSH-style Trust On First Use. Approve a peer once, reconnections are automatic. Each peer pair has its own unique derived key.

⚡

Instant discovery

mDNS finds peers within seconds of startup. Auto-discovery or manual dial by PeerId — your choice.

📦

Chunked transfers

Files are split, BLAKE3-hashed, and verified chunk by chunk. Interrupted transfers stall-detect and re-request missing pieces.

🦀

Written in Rust

libp2p, tokio, ChaCha20-Poly1305, BLAKE3. A single static binary — no runtime, no interpreter, no shared library dependencies.

How it works

Five steps from cold start to syncing.

1

Start on each machine

Run the binary — it starts a GUI on port 9000 and begins broadcasting via mDNS multicast.

2

Peers discover each other

mDNS reaches every device on your subnet within seconds. No manual IP entry needed, though manual dial is supported too.

3

Approve on first connect

A TOFU prompt appears once per new peer. After approval, future reconnects are fully automatic.

4

Keys derived, never shared

X25519 Diffie-Hellman derives a unique shared key per peer pair. Keys live only in ~/.vitruvius/ with 0600 permissions.

5

Set your folder and sync

Point Vitruvius at a folder. Files are watched in real time. Any change syncs to all approved peers immediately.

Download

Static binaries — no runtime dependencies, no installer needed.

One-line install — downloads a fully static binary, installs to /usr/local/bin, and registers a systemd user service that starts on login.

Install (x86_64 Linux)

curl -fsSL https://github.com/XXNOUR/Vitruvius/raw/main/install.sh | bash

Or grab just the binary:

Manual download

curl -fsSL https://github.com/XXNOUR/Vitruvius/releases/download/v0.3.0/vitruvius-linux-x86_64 -o vitruvius && chmod +x vitruvius && sudo mv vitruvius /usr/local/bin/

After install

systemctl --user status vitruvius# check if running

systemctl --user stop vitruvius# stop

systemctl --user restart vitruvius# restart

journalctl --user -u vitruvius -f# live logs

Quick start

Typical first-run flow for two machines on the same network.

1. Start on both machines

vitruvius

GUI opens at http://localhost:9000. Your PeerId is printed at startup.

2. Peers appear automatically

mDNS discovers peers on the same subnet within seconds. If they don't appear (firewall, VPN, etc.), click Dial and enter the peer's IP address.

3. Approve once

A dialog appears the first time. After approval, an X25519 key is derived and stored in ~/.vitruvius/ with 0600 permissions. All future reconnects skip the dialog.

4. Set your folder

Enter a folder path in the GUI. Files are watched in real time — anything added or removed syncs immediately to all approved peers.

CLI flags

--http-port <PORT>GUI HTTP port (default: 9000)
--ws-port <PORT>WebSocket port (default: 9001)
--node-name <NAME>Override this node's display name
--key-path <FILE>Use a pre-shared key file instead of TOFU
--generate-key <FILE>Generate a new 32-byte key file and exit

Security model

SSH-style TOFU. No PKI, no CA, no certificates to manage.

✅ Encrypted in transit

ChaCha20-Poly1305 authenticated encryption. Any tampered byte causes decryption to fail.

✅ Keys never leave your LAN

X25519 DH happens directly peer-to-peer. No server ever sees key material.

✅ Per-peer unique keys

Each peer pair derives its own shared key. Compromising one pair doesn't affect others.

✅ Stable identity

Ed25519 keypair persists in ~/.vitruvius/ — your PeerId is consistent across restarts.

⚠️ First-connect TOFU risk

Like SSH, a MITM present on the very first handshake can intercept keys. Compare fingerprints out-of-band for stronger guarantees.

⚠️ LAN-only by design

mDNS doesn't cross routers. Vitruvius is built for trusted local networks, not internet-facing use.
